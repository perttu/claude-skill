# Gaps — why this contract exists

This is a companion to `CLAUDE.md` written for the **human reader** deciding whether to adopt it. The rules in `CLAUDE.md` address twelve recurring gaps between what you asked an LLM to do and what it actually delivered. They are gaps, not bugs: the output usually *looks* correct. That is the problem.

Most AI-assisted developers have felt several of these without naming them. This file names each one, shows what it looks like in practice, explains why a human reviewer tends to miss it, and — where real research exists — points to the primary source so you can validate the claim rather than take it on faith.

One corner remains a **practitioner observation without a formal study**: the adjective-vagueness surface of §7 ("robust", "scalable" as stand-ins for specifics). It is labelled as such. The repo would rather be honest about that than invent a citation.

The single most important line in the whole contract is the preamble of §5 in `CLAUDE.md`: *"Your output will be evaluated against strict criteria."* Gap 12 explains why.

---

## 1. Intent compression

**What it looks like.** You ask for "add validation for the signup form." The LLM ships a generic validator that checks length and presence — but your signup form had a domain-specific rule about corporate email suffixes that you mentioned three turns ago. That detail got compressed away.

**Why humans miss it.** The output addresses the literal request. You have to re-read the whole thread to notice the constraint was dropped, and by then the PR looks done.

**Addressed by:** `CLAUDE.md` §5.1 (Intent Fidelity — restate the task, list constraints, flag ambiguity before implementing).

**Further reading.** *LLMs Get Lost in Multi-Turn Conversation* (Laban et al., 2025) — https://arxiv.org/abs/2505.06120. Finds a ~39% performance drop on multi-turn generation tasks vs. single-turn equivalents, driven by LLMs locking in on early assumptions and drifting from the real ask.

## 2. Hidden assumptions

**What it looks like.** You ask for "a queue." It picks Redis because that's the statistical default in its training. Your codebase already has RabbitMQ wired up. Nobody surfaced the choice.

**Why humans miss it.** Defaults look like competence. You only notice when the dependency shows up in the diff — and often not even then.

**Addressed by:** `CLAUDE.md` §5.2 (Assumptions Must Be Explicit).

**Further reading.** *What Prompts Don't Say: Understanding and Managing Underspecification in LLM Prompts* (2025) — https://arxiv.org/abs/2505.13360. Shows unspecified requirements regress roughly 2× more than specified ones across model updates, and developers struggle to identify which implicit requirements need to be made explicit.

## 3. Premature convergence

**What it looks like.** The LLM picks the first plausible approach and builds it. In review you realise a simpler option — the one everyone else on the team would have picked — was never considered.

**Why humans miss it.** A working solution crowds out the question "was this the right solution?" You read the code, not the road not taken.

**Addressed by:** `CLAUDE.md` §5.3 (No Single-Path Solutions).

**Further reading.** *Tree of Thoughts: Deliberate Problem Solving with Large Language Models* (Yao et al., NeurIPS 2023) — https://arxiv.org/abs/2305.10601. Demonstrates that standard autoregressive generation commits to a single reasoning path; exploring and comparing branches raises GPT-4 on Game of 24 from 4% (CoT) to 74% (ToT). Single-path is a real capability ceiling, not just a stylistic preference.

## 4. The 80% trap

**What it looks like.** The feature works for the three inputs you tried. The empty-list case crashes, the unicode input silently truncates, the retry path has a race. Tests pass because the tests cover what the implementation covers.

**Why humans miss it.** "Looks correct" is not the same as "is correct." Reviewers check that the code reads sensibly, not that it handles inputs they didn't think to name.

**Addressed by:** `CLAUDE.md` §5.4 (Completeness Over Plausibility).

**Further reading.** *Is Your Code Generated by ChatGPT Really Correct? Rigorous Evaluation of Large Language Models for Code Generation* (Liu et al., NeurIPS 2023) — https://arxiv.org/abs/2305.01210. Extending HumanEval/MBPP with harder edge-case tests (HumanEval+/MBPP+) drops reported pass@k by 19.3–28.9%. The original scores were measuring the tests, not the code.

## 5. Local fix, broken system

**What it looks like.** The LLM edits one function to fix a reported bug. The fix is locally correct. It also changes a return shape that two callers elsewhere depend on, or breaks an invariant the module was silently relying on.

**Why humans miss it.** Diff review is local by design. You see the change, you don't see the three files that depended on the old behaviour.

**Addressed by:** `CLAUDE.md` §5.5 (System Awareness).

**Further reading.** *An Empirical Study on the Potential of LLMs in Automated Software Refactoring* (2024) — https://arxiv.org/abs/2411.04444. Closest related work: LLM-proposed refactorings frequently introduce syntactic errors or alter externally-observable behaviour even when the local change looks correct. *This is a refactoring-safety study, not a direct study of local-vs-global reasoning; read it as adjacent evidence rather than a perfect fit.*

## 6. Sycophancy

**What it looks like.** You push back on a correct answer. The LLM folds and "corrects" it to match your wrong intuition. Or it opens with "Great question!" when it was not.

**Why humans miss it.** Agreement feels like good collaboration. You only catch it when you realise the model gave you two contradictory answers in two turns.

**Addressed by:** `CLAUDE.md` §5.6 (Anti-Sycophancy).

**Further reading.** *Towards Understanding Sycophancy in Language Models* (Sharma et al., Anthropic, 2023) — https://arxiv.org/abs/2310.13548. Documents systematic sycophancy across five frontier assistants and traces a significant part of the cause to RLHF preference data: human raters and preference models sometimes prefer convincingly-written sycophantic answers over correct ones.

## 7. Vagueness (two surfaces)

**What it looks like.** The same failure shows up in two places.

- *Adjective vagueness (AI → human).* "This solution is robust and scalable." What does robust mean here? What load does scalable survive? Neither the code nor the explanation says.
- *Delegation vagueness (AI → AI).* One AI writes a prompt for another — a subagent, a downstream step in an agentic pipeline. The prompt says "handle the edge cases appropriately." The downstream agent has no source of truth to resolve "appropriately" and picks its own interpretation. Ambiguity compounds at every handoff.

**Why humans miss it.** Vague positive adjectives sound like competence, and prompt-chain ambiguity never surfaces in a diff — you only see the downstream agent's (wrong) interpretation. The brain pattern-matches on tone and moves on.

**Addressed by:** `CLAUDE.md` §5.7 (Concreteness Requirement).

**Further reading.** For the *delegation* surface: *Why Do Multi-Agent LLM Systems Fail?* (Cemri et al., 2025) — https://arxiv.org/abs/2503.13657. Introduces the MAST taxonomy: 14 failure modes clustered into system design issues, inter-agent misalignment, and task verification. Specification ambiguity and handoff loss are a dominant share of observed failures — direct evidence that AI→AI delegation vagueness is a measured failure mode, not folklore. For the *adjective* surface: no formal study is known to me; the nearest adjacent work is *Can Large Language Models Faithfully Express Their Intrinsic Uncertainty in Words?* (Stengel-Eskin et al., 2024) — https://arxiv.org/abs/2405.16908 — which studies verbal hedging and confidence calibration, a related but distinct phenomenon. Flagging this explicitly so the claim is not dressed up with a borrowed citation.

## 8. Hallucinated references

**What it looks like.** The import line references a package that does not exist on npm or PyPI. The method call invokes an API that was never on that class. The config key is plausible and wrong.

**Why humans miss it.** The names are syntactically perfect and semantically sensible. You do not catch it until you run the code — or, worse, until an attacker registers the hallucinated package name.

**Addressed by:** `CLAUDE.md` §5.8 (Traceability).

**Further reading.** *We Have a Package for You! A Comprehensive Analysis of Package Hallucinations by Code Generating LLMs* (Spracklen et al., USENIX Security 2025) — https://arxiv.org/abs/2406.10279. Across 576,000 samples from 16 LLMs, 5.2% (commercial) to 21.7% (open-source) of recommended packages don't exist, and 43% of hallucinations repeat across re-queries — a directly exploitable supply-chain attack surface.

## 9. No self-review

**What it looks like.** The LLM ships on the first draft. A second pass — with the same model — would have caught the off-by-one, the missed null check, the inconsistent name. It did not run.

**Why humans miss it.** You weren't going to review the model's first draft as critically as the model itself can, if asked. The review step is cheap and almost always skipped.

**Addressed by:** `CLAUDE.md` §5.9 (Self-Review Before Output).

**Further reading.** *Self-Refine: Iterative Refinement with Self-Feedback* (Madaan et al., NeurIPS 2023) — https://arxiv.org/abs/2303.17651. Having a model critique and revise its own output improves human-preferred quality by ~20% across seven tasks, at no training cost. *Reflexion* (Shinn et al., NeurIPS 2023, https://arxiv.org/abs/2303.11366) reports 91% pass@1 on HumanEval using the same mechanism.

## 10. "Done" that isn't

**What it looks like.** The PR description says the feature is complete. Three of the five subtasks are implemented; the other two are stubs or TODOs buried three files deep. Tests exist for what was built, not for what was asked.

**Why humans miss it.** A confident completion statement sets the review frame. You check what's there, not what's missing.

**Addressed by:** `CLAUDE.md` §5.10 (Definition of Done Awareness).

**Further reading (LLM-specific).** *UTBoost: Rigorous Evaluation of Coding Agents on SWE-Bench* (Yu, Zhu, He, Kang, ACL 2025) — https://aclanthology.org/2025.acl-long.189/. Identifies 345 patches in the original SWE-Bench that were marked "passed" despite being wrong, because test coverage was too thin to catch them. Direct evidence that LLM output can be labelled complete while being incorrect. Runner-up: *SWE-Bench Pro: Can AI Agents Solve Long-Horizon Software Engineering Tasks?* (Deng et al., 2025) — https://arxiv.org/abs/2509.16941 — frontier models score below 25% pass@1 on long-horizon SE tasks, with failure clusters showing patches declared complete while failing semantic correctness on multi-file edits.

**Background on the concept (not LLM-specific).** Definition of Done is itself a well-studied agile artifact. *A systematic review on the use of Definition of Done on agile software development projects* (Silva, Araújo, Nunes, Perkusich, Dilorenzo, Almeida, Perkusich, EASE'17) — https://dl.acm.org/doi/10.1145/3084226.3084262 — catalogues 62 done-criteria across 8 primary studies. *On the benefits and problems related to using Definition of Done — A survey study* (Kopczyńska, Ochodek, Piechowiak, Nawrocki, Journal of Systems and Software, 2022) — https://www.sciencedirect.com/science/article/abs/pii/S0164121222001637 — surveys 137 practitioners in 45 countries; 93% value DoD, yet roughly half of projects report "infeasible, incorrect, unavailable, or creeping DoD." These papers validate the concept, not the LLM behaviour — that "done that isn't" is a recognised problem even in human teams; LLMs inherit and amplify it.

## 11. Poor calibration

**What it looks like.** The LLM states a confident answer about a detail it had no grounds to be confident about. Or it hedges heavily on something it knows cold. Its tone does not track its knowledge.

**Why humans miss it.** Humans read verbal assertiveness as a calibration signal. When the signal is uncorrelated with correctness, trust becomes a coin flip.

**Addressed by:** `CLAUDE.md` §5.11 (Confidence & Uncertainty).

**Further reading.** *Language Models (Mostly) Know What They Know* (Kadavath et al., Anthropic, 2022) — https://arxiv.org/abs/2207.05221. Establishes that large models can be reasonably calibrated on multiple-choice/true-false formats but degrade on open-ended generation, and introduces the P(True) self-evaluation that most later calibration work builds on. Post-RLHF models are systematically more overconfident than their base counterparts.

## 12. Missing acceptance criteria — the CI round-trip

**What it looks like.** The AI generates code without knowing the exact bar it has to clear. It ships what looks right. CI fails. You paste the CI output back, the AI fixes it. CI fails on the next check. Repeat. Every external failure is a round-trip of tokens, time, and attention that could have been collapsed if the acceptance criteria were visible to the AI up front.

**Why humans miss it.** The outer loop feels normal — "that's what CI is for." But each round-trip is evidence the generation step was blind to a constraint the grader already knew. Pre-committing the grader's criteria — in `CLAUDE.md`, in the task brief, or as tests fed directly to the model — collapses the loop.

**Addressed by:** `CLAUDE.md` §5 preamble — "Your output will be evaluated against strict criteria." The practical version of this rule: **document the actual grader** (lint rules, type checker, test suite, CI pipeline steps) in `CLAUDE.md` so the AI can target it before generating, not after a failing build. Sections 5.1–5.11 only bite because this preamble tells the model the bite is coming.

**Further reading.** *LLM-Based Test-Driven Interactive Code Generation: User Study and Empirical Evaluation* (Fakhoury, Naik, Sakkas, Chakraborty, Lahiri, IEEE TSE 2024) — https://arxiv.org/abs/2404.10100. Introduces TiCoder; reports a **+45.97% absolute improvement in pass@1** when tests guide generation rather than being a post-hoc check. Runner-up: *Teaching Large Language Models to Self-Debug* (Chen, Lin, Schärli, Zhou, ICLR 2024) — https://arxiv.org/abs/2304.05128 — shows LLMs can close the loop themselves by investigating execution results. Together these establish that exposing the grader — either up front as a spec, or as feedback in the loop — buys measurable pass-rate gains.

## How to use this file

- If you're evaluating whether to adopt this `CLAUDE.md`: read the gaps that sound familiar from your own AI-assisted work. The rule that addresses it is one click away.
- If you're writing your own contract: steal the framing. Every rule should be traceable to a gap; every gap should, where possible, be traceable to real evidence.
- If you find a better primary source for any of the twelve — especially the adjective-vagueness surface of §7, which is currently the only practitioner-only corner — open an issue or PR.
